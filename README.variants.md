# Варианты API: match, lot/analyze, match/analyze

Базовый URL API: `http://127.0.0.1:8083`. Профиль в теле можно не передавать, если задан в `.env`.

```bash
curl -s http://127.0.0.1:8083/health
```

Для **2** и **4** лоты должны быть в базе — сначала индексация (для **3** не обязательна):

```bash
curl -s -o /dev/null -w "HTTP:%{http_code}\n" \
  -X POST "http://127.0.0.1:8083/v1/lots/50225530/index" \
  -H "Content-Type: application/json" \
  -d '{"text":"Полный или краткий текст лота / ТЗ здесь","source_hint":"api"}'
```

---

## 2 — `POST /v1/match`

- **Только локальная модель + база (pgvector).** OpenAI не используется.
- Берёт **профиль компании** (из запроса или из `.env`) и ищет по **уже проиндексированным** текстам в БД.
- **Ответ:** список `lot_id`, числовой `score`, `snippets` (куски ТЗ из базы).
- **Нет** готовых фраз «подходит / не подходит» — только **похожесть** и **цитаты**.

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match \
  -H "Content-Type: application/json" \
  -d '{"top_lots":5,"chunk_hits":40,"snippets_per_lot":2}'
```

С явным профилем:

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match \
  -H "Content-Type: application/json" \
  -d '{"profile":"хостинг, ЦОД, облако, Казахстан","top_lots":5}'
```

## 3 — `POST /v1/lot/analyze`

- **Только OpenAI.** База для поиска не участвует (лот не обязан быть в индексе).
- В теле один **`lot_text`** (+ профиль из запроса или `.env`).
- **Ответ:** `summary`, `fit` (подходит / сомнительно / не подходит), `reason`, `checks`.
- Удобно: *«вот текст одного лота — скажи по-человечески»*, без заранее загонять лот в RAG.

Нужен `OPENAI_API_KEY` в `.env`.

```bash
curl -s --max-time 120 -X POST http://127.0.0.1:8083/v1/lot/analyze \
  -H "Content-Type: application/json" \
  -d '{"lot_text":"Услуги хостинга, продление. Астана, сумма 50000 тг."}'
```

---

## 4 — `POST /v1/match/analyze`

- **И то и другое:** сначала как в п. **2** — поиск по базе (кто из сохранённых лотов ближе к профилю), потом OpenAI читает профиль + найденные `snippets` (и опционально `incoming`) и пишет `summary` + `verdicts` по каждому из топ-лотов.
- **Ответ:** и структурированный разбор, и `matches` (как в п. 2).
- Удобно: *«по всей нашей базе кто в топе и почему словами».*

Нужен `OPENAI_API_KEY` в `.env`.

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match/analyze \
  -H "Content-Type: application/json" \
  -d '{"top_lots":5,"snippets_per_lot":2}'
```

С дополнительным контекстом лота (опционально):

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match/analyze \
  -H "Content-Type: application/json" \
  -d '{
    "top_lots": 5,
    "snippets_per_lot": 2,
    "incoming": "{\"id\":50225530,\"title\":\"...\",\"description\":\"...\"}"
  }'
```

---

## Одной таблицей

| | База (индекс) | OpenAI | Вход «один лот текстом» |
|--|:--:|:--:|:--|
| **2** `match` | да | нет | нет (только то, что уже в БД) |
| **3** `lot/analyze` | нет | да | да |
| **4** `match/analyze` | да | да | опционально (`incoming`) |

**Итог:** **2** — чистый «умный поиск по базе»; **3** — «разбери один лот без базы»; **4** — «поиск по базе + человеческий вердикт».

Первый запрос после перезапуска API (особенно **2** / **4**) иногда долгий — не уменьшай `--max-time` без нужды.

# Индексация лота — `POST /v1/lots/{lot_id}/index`

## Зачем нужен этот эндпоинт

Чтобы текст лота участвовал в **семантическом поиске** по Postgres (pgvector). Без записи в базу лот **не появится** в выдаче **`POST /v1/match`** и **`POST /v1/match/analyze`**.

Это **не** про OpenAI: только локальная модель эмбеддингов и сохранение чанков в БД.

## Кому обязателен

| Сценарий | Нужен `/index`? |
|----------|-----------------|
| Вариант 2 — `POST /v1/match` | да, для каждого лота, который должен искаться |
| Вариант 4 — `POST /v1/match/analyze` | да, иначе лот не попадёт в топ из базы |
| Вариант 3 — `POST /v1/lot/analyze` | **нет**, туда текст передаётся в `lot_text` |

## Поведение

- **`lot_id`** в пути — тот же идентификатор, что у лота у вас (например `50225530`).
- **Тело:** `{"text": "…полный или краткий текст лота / ТЗ…", "source_hint": "опционально"}`.
- **Ответ:** обычно **HTTP 204** (без тела).
- Текст лота **изменился** — снова вызовите `index` с тем же `lot_id` и новым `text` (старые чанки для этого лота заменяются).

## Порядок работы с вариантами 2 и 4

1. **`POST /v1/lots/{lot_id}/index`** для нужных лотов.
2. Затем **`POST /v1/match`** или **`POST /v1/match/analyze`**.

## Примеры `curl`

Базовый URL: `http://127.0.0.1:8083`.

Проверка сервиса:

```bash
curl -s http://127.0.0.1:8083/health
```

Индексация (успех — код **204**):

```bash
curl -s -o /dev/null -w "HTTP:%{http_code}\n" \
  -X POST "http://127.0.0.1:8083/v1/lots/50225530/index" \
  -H "Content-Type: application/json" \
  -d '{"text":"Полный или краткий текст лота / ТЗ здесь","source_hint":"api"}'
```

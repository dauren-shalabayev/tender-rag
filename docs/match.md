# Вариант 2 — `POST /v1/match`

## Что делает

- **Только локальная модель эмбеддингов + база (pgvector).** OpenAI **не** используется.
- Берёт **профиль компании** (из тела запроса или из `.env`: `COMPANY_PROFILE` / `COMPANY_PROFILE_FILE`) и ищет по **уже проиндексированным** текстам в БД.
- Возвращает **ранжированный топ** лотов по **похожести** на профиль.

## Что в ответе

- Список **`lot_id`**, числовой **`score`**, **`snippets`** — фрагменты ТЗ из базы.
- **Нет** готовых формулировок «подходит / не подходит» — только **сходство** и **цитаты** для своей логики на фронте/бэке.

## Требования

- Лоты должны быть заранее записаны: [`POST /v1/lots/{lot_id}/index`](indexing.md).
- Первый запрос после перезапуска API может быть долгим (загрузка модели эмбеддингов).

## Когда использовать

Массовый или регулярный отбор: «из **наших** сохранённых лотов кто ближе всего к профилю» **без** стоимости и задержек LLM.

## Сводка по режимам

| База (индекс) | OpenAI | Один лот текстом в запросе |
|:--:|:--:|:--:|
| да | нет | нет (только то, что уже в БД) |

## Примеры `curl`

`http://127.0.0.1:8083` — при необходимости замените на свой хост.

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match \
  -H "Content-Type: application/json" \
  -d '{"top_lots":5,"chunk_hits":40,"snippets_per_lot":2}'
```

С явным профилем в теле:

```bash
curl -s --max-time 300 -X POST http://127.0.0.1:8083/v1/match \
  -H "Content-Type: application/json" \
  -d '{"profile":"хостинг, ЦОД, облако, Казахстан","top_lots":5}'
```

Не уменьшайте `--max-time` без нужды на «первом» запросе после старта.
